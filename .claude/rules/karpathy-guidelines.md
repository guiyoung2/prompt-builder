# Karpathy 행동 지침

LLM이 코딩에서 자주 저지르는 실수를 줄이기 위한 행동 지침. 필요에 따라 프로젝트별 지침과 함께 적용한다.

**트레이드오프:** 이 지침은 속도보다 신중함에 무게를 둔다. 사소한 작업은 상황에 맞게 판단할 것.

## 1. 코드 작성 전에 먼저 생각하기

**가정하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러낼 것.**

구현 전에:
- 가정을 명시적으로 말할 것. 확실하지 않으면 질문할 것.
- 해석이 여러 가지 가능하다면 모두 제시할 것 — 혼자 임의로 고르지 말 것.
- 더 간단한 접근이 있다면 말할 것. 필요할 때는 반대 의견도 제시할 것.
- 불명확한 부분이 있으면 멈출 것. 무엇이 혼란스러운지 명시하고 질문할 것.

## 2. 단순함 우선

**문제를 해결하는 최소한의 코드만 작성. 추측성 코드는 금지.**

- 요청되지 않은 기능은 추가하지 말 것.
- 한 번만 쓰이는 코드에 추상화를 만들지 말 것.
- 요청되지 않은 "유연성"이나 "설정 가능성"은 만들지 말 것.
- 일어날 수 없는 상황에 대한 에러 처리는 하지 말 것.
- 200줄을 작성했는데 50줄로 가능하다면 다시 쓸 것.

스스로에게 물을 것: "시니어 엔지니어가 이걸 보면 과하다고 할까?" 그렇다면 단순화할 것.

## 3. 외과적인(최소한의) 변경

**꼭 건드려야 하는 것만 건드릴 것. 자기가 만든 흔적만 정리할 것.**

기존 코드를 수정할 때:
- 인접한 코드, 주석, 포맷팅을 임의로 "개선"하지 말 것.
- 멀쩡하게 동작하는 코드를 리팩터링하지 말 것.
- 본인 스타일과 다르더라도 기존 스타일을 따를 것.
- 무관한 데드 코드(쓰이지 않는 코드)를 발견하면 언급만 하고 삭제하지 말 것.

본인의 변경으로 고아(orphan) 코드가 생긴 경우:
- 본인의 변경으로 인해 더 이상 쓰이지 않게 된 import, 변수, 함수는 제거할 것.
- 요청받지 않은 한 기존 데드 코드는 제거하지 말 것.

판단 기준: 변경된 모든 라인은 사용자의 요청과 직접적으로 연결되어야 한다.

## 4. 목표 기반 실행

**성공 기준을 정의하고, 검증될 때까지 반복할 것.**

작업을 검증 가능한 목표로 변환할 것:
- "검증 추가" → "잘못된 입력에 대한 테스트를 작성하고 통과시킬 것"
- "버그 수정" → "버그를 재현하는 테스트를 먼저 작성하고 통과시킬 것"
- "X 리팩터링" → "리팩터링 전후 모두 테스트가 통과하는지 확인할 것"

여러 단계로 이루어진 작업은 간단한 계획을 먼저 제시할 것:

```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

명확한 성공 기준이 있어야 독립적으로 루프를 돌며 작업할 수 있다. 기준이 약하면("그냥 동작하게 해줘") 계속 사용자에게 다시 물어봐야 한다.
